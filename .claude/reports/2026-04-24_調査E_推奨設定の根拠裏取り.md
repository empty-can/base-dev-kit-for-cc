# 調査E: 推奨設定の根拠裏取り

## 調査概要

**調査目的**: Claude Code ベースキットで採用予定の推奨設定について、公式ドキュメントから根拠を引用付きで整理し、「なぜそれがデファクトか」を裏付ける

**対象項目数**: 12 項目

**実施日**: 2026-04-24

## 設定項目ごとの根拠

### 項目1: permissions の deny パターン

**推奨内容**: `.env` / `secrets/**` / `~/.aws/credentials` の Read を deny

**根拠（引用）**:
- https://code.claude.com/docs/en/permissions
- > 「Exclude Sensitive Files ... "deny": [ "Read(./.env)", "Read(./.env.*)", "Read(./secrets/**)", "Read(./config/credentials.json)" ]」
- > 「A Read(./.env) deny rule blocks the Read tool but does not prevent cat .env in Bash. For OS-level enforcement that blocks all processes from accessing a path, enable the sandbox.」

**裏取り充足度**: 強い（公式 Common Configuration Examples で明記）

**注意**: Bash 経由の cat には効かないため、sandbox 併用推奨

### 項目2: permissions の allow パターン

**推奨内容**: 頻出コマンド（`npm run *`, `git *`）を allow

**根拠（引用）**:
- https://code.claude.com/docs/en/settings
- > 「"allow": ["Bash(npm run *)", "Bash(git *)"], "deny": ["Bash(curl *)"]」
- https://code.claude.com/docs/en/permissions
- > 「Bash permission rules support wildcard matching with `*`」
- > 「Bash permission patterns that try to constrain command arguments are fragile」

**裏取り充足度**: 強い

**注意**: allow パターンはプロジェクト依存。汎用ベースキットでは最小限に留めるべき。

### 項目3: CLAUDE.md の推奨サイズ

**推奨内容**: 200 行以内を target とする

**根拠（引用）**:
- https://code.claude.com/docs/en/memory
- > 「Size: target under 200 lines per CLAUDE.md file. Longer files consume more context and reduce adherence.」
- > 「CLAUDE.md files are loaded into the context window at the start of every session, consuming tokens alongside your conversation.」
- > 「shorter files produce better adherence」

**裏取り充足度**: 強い

### 項目4: CLAUDE.md のファイル配置

**推奨内容**: `./CLAUDE.md` または `./.claude/CLAUDE.md`（どちらも等価）

**根拠（引用）**:
- https://code.claude.com/docs/en/memory
- > 「Project instructions | `./CLAUDE.md` or `./.claude/CLAUDE.md` | Team-shared instructions for the project」
- > 「A project CLAUDE.md can be stored in either `./CLAUDE.md` or `./.claude/CLAUDE.md`.」
- > 「For private per-project preferences... create a `CLAUDE.local.md`. Add `CLAUDE.local.md` to your `.gitignore`」

**裏取り充足度**: 強い

### 項目5: settings.json の階層構造

**推奨内容**: Managed > CLI > Local > Project > User の優先度

**根拠（引用）**:
- https://code.claude.com/docs/en/settings
- > 「Precedence (highest to lowest): 1. Managed settings 2. Command line arguments 3. Local project settings 4. Shared project settings 5. User settings」
- > 「Array settings merge across scopes (concatenated, deduplicated)」

**裏取り充足度**: 強い

### 項目6: CLAUDE.local.md / settings.local.json を gitignore する理由

**推奨内容**: 個人設定・環境依存設定は git に commit しない

**根拠（引用）**:
- https://code.claude.com/docs/en/memory
- > 「For private per-project preferences that shouldn't be checked into version control, create a `CLAUDE.local.md` at the project root. Add `CLAUDE.local.md` to your `.gitignore`」
- https://code.claude.com/docs/en/settings
- > 「.claude/settings.local.json | You, this repo only | No (gitignored)」

**裏取り充足度**: 強い

### 項目7: hooks の各イベント用途

**推奨内容**: SessionStart / PreToolUse / PostToolUse / Stop など用途別イベント

**根拠（引用）**:
- https://code.claude.com/docs/en/hooks-guide
- > 「Hooks are user-defined shell commands that execute at specific points in Claude Code's lifecycle. They provide deterministic control over Claude Code's behavior, ensuring certain actions always happen rather than relying on the LLM to choose to run them.」
- https://code.claude.com/docs/en/permissions
- > 「When Claude Code makes a tool call, PreToolUse hooks run before the permission prompt. The hook output can deny the tool call, force a prompt, or skip the prompt」

**裏取り充足度**: 中程度（詳細な use case list は full reference 参照が必要）

### 項目8: sub-agents の設計指針

**推奨内容**: Context flooding を避けるため、専門的なサブタスクに使う

**根拠（引用）**:
- https://code.claude.com/docs/en/sub-agents
- > 「Use one when a side task would flood your main conversation with search results, logs, or file contents you won't reference again: the subagent does that work in its own context and returns only the summary.」
- > 「Subagents help you: Preserve context / Enforce constraints / Reuse configurations / Specialize behavior / Control costs」
- > 「When you create a subagent, write a clear description so Claude knows when to use it.」

**裏取り充足度**: 強い

### 項目9: slash commands の設計指針

**推奨内容**: Custom command は skill と等価

**根拠（引用）**:
- https://code.claude.com/docs/en/skills
- > 「Custom commands have been merged into skills. A file at `.claude/commands/deploy.md` and a skill at `.claude/skills/deploy/SKILL.md` both create `/deploy` and work the same way.」

**裏取り充足度**: 強い

**含意**: 本ベースキットでは skills 側で実装する方が将来性が高い。

### 項目10: skills の設計指針

**推奨内容**: 繰り返し手順を skill 化、CLAUDE.md は fact のみに

**根拠（引用）**:
- https://code.claude.com/docs/en/skills
- > 「Create a skill when you keep pasting the same playbook, checklist, or multi-step procedure into chat, or when a section of CLAUDE.md has grown into a procedure rather than a fact.」
- > 「Keep `SKILL.md` under 500 lines.」
- > 「disable-model-invocation: true: Only you can invoke the skill. Use this for workflows with side effects...like /commit, /deploy」

**裏取り充足度**: 強い

### 項目11: sandbox の有効化推奨

**推奨内容**: permission prompt fatigue を避けたい場合に有効化

**根拠（引用）**:
- https://code.claude.com/docs/en/sandboxing
- > 「Sandboxing addresses these challenges by: 1. Defining clear boundaries 2. Reducing permission prompts 3. Maintaining security 4. Enabling autonomy」
- > 「Approval fatigue: Repeatedly clicking 'approve' can cause users to pay less attention to what they're approving」
- > 「Effective sandboxing requires both filesystem and network isolation.」

**裏取り充足度**: 強い

### 項目12: プラグイン vs カスタム設定の使い分け

**推奨内容**: 個人・プロジェクト固有なら標準 `.claude/`、複数プロジェクト・チーム共有なら plugin

**根拠（引用）**:
- https://code.claude.com/docs/en/plugins
- > 「Use standalone configuration when: customizing for a single project / personal / experimenting / want short skill names like `/hello`」
- > 「Use plugins when: share with team or community / same skills/agents across multiple projects / version control / distributing through marketplace」
- > 「Start with standalone configuration in `.claude/` for quick iteration, then convert to a plugin when you're ready to share.」

**裏取り充足度**: 強い

**含意**: 本ベースキットは「自分の新規プロジェクトのスタート地点」なので **標準 `.claude/` 構成で進める**（plugin 化は不要）。

## ベースキット設計への含意

### 確定判断
1. `.claude/` 配下に CLAUDE.md, settings.json を配置（plugin 化しない）
2. `.env` / `secrets` / AWS 認証情報の deny は公式例そのまま採用
3. `.gitignore` に `CLAUDE.local.md` および `.claude/settings.local.json` を必ず含める
4. custom commands は **skills として実装** する（将来性のため）
5. sub-agents サンプルは含める（コンテキスト管理のベストプラクティス）

### 未確定事項
1. hooks のクロスプラットフォーム実装方針（bash+PowerShell vs Node.js など）
2. sandbox のデフォルト挙動（有効化 / 無効化 / ガイドのみ）

これらは最終計画書策定時に決定する。
