## CRITICAL（事実誤り・要修正）

### C-01: 「plugin.json（必須）」は事実誤り

- **箇所**: 1-2「Plugin の構造」図中コメント（L41）
- **記載内容**: `│   └── plugin.json          # プラグインマニフェスト（必須）`
- **問題点**: 公式仕様（plugins-reference）では plugin.json は **optional**
- **公式記述**: *"The manifest is optional. If omitted, Claude Code auto-discovers components in default locations and derives the plugin name from the directory name. Use a manifest when you need to provide metadata or custom component paths."*
- **修正案**: 「プラグインマニフェスト（任意。省略時はディレクトリ名がプラグイン名になる）」

### C-02: 代替手段 C の前提が技術的に誤り

- **箇所**: 3-1「比較対象とする代替手段」表（L303）
- **記載内容**: 「CLAUDE.md + rules/ の参照リポジトリ方式 — 共通設定を別リポジトリで管理し、`CLAUDE.md` に `@<url>` インポートで参照」
- **問題点**: CLAUDE.md の `@` インポートは**ローカルファイルパスのみ**で、URL インポートはサポートされていない
- **公式記述**（memory ページ）: *"CLAUDE.md files can import additional files using `@path/to/import` syntax. Both relative and absolute paths are allowed. Relative paths resolve relative to the file containing the import, not the working directory."*
- **修正案**: 代替手段 C を削除するか、「git submodule で取り込んだローカルパスを `@` インポート」に修正

## IMPORTANT（重要な改善点）

### I-01: CLAUDE_PLUGIN_DATA の削除タイミングが不正確

- **箇所**: 3-2 (5)「永続データディレクトリ」（L388）
- **記載内容**: 「プラグインのアンインストール時に自動削除される」
- **問題点**: 厳密には「最後のスコープからアンインストール時」
- **公式記述**: *"The data directory is deleted automatically when you uninstall the plugin from **the last scope** where it is installed."*
- **修正案**: 「最後のスコープからアンインストールされたとき自動削除される」

### I-02: 移行手順 ④ が不完全（実行できない）

- **箇所**: 2-3「既存の Standalone 設定を Plugin に移行する手順」④（L264-265）
- **記載内容**:
  ```bash
  # ④ hooks を移行（settings.json の "hooks" オブジェクトを抽出して hooks/hooks.json へ）
  mkdir my-plugin/hooks
  ```
- **問題点**: `mkdir` のみで実コピーコマンドがコメントだけ。手順としてコピペ実行できない
- **修正案**: 具体的なコマンド例を追加
  ```bash
  mkdir my-plugin/hooks
  jq '{hooks: .hooks}' .claude/settings.json > my-plugin/hooks/hooks.json
  ```

### I-03: marketplace.json 例で `pluginRoot` の効果が示されていない

- **箇所**: 1-3「marketplace.json の構造」例（L113-118）
- **問題点**: `metadata.pluginRoot: "./plugins"` を設定しているのに、各プラグインの `source` は `"./plugins/code-formatter"` のフルパス指定。pluginRoot の効果が現れていない
- **修正案**: pluginRoot を活用する形に変える
  ```json
  "metadata": { "pluginRoot": "./plugins" },
  "plugins": [
    { "name": "code-formatter", "source": "code-formatter", ... }
  ]
  ```
  または `pluginRoot` を削除して現状の source パスのままにする

### I-04: Step 4 で settings.json の配置先が不明瞭

- **箇所**: 2-1 Step 4「プロジェクト設定でチームに自動配布する」（L211）
- **記載内容**: 「`.claude/settings.json` に追加することで…」
- **問題点**: どのリポジトリの settings.json かが明示されていない。利用側プロジェクトに書くべきだが、Marketplace リポジトリ側と誤解されうる
- **修正案**: 「**利用側プロジェクトの** `.claude/settings.json` に追加」と明記

### I-05: スコープ優先順位から `Command line` が脱落

- **箇所**: 2-2「インストールスコープの選択」（L235）
- **記載内容**: 「優先順位（高→低）: managed > local > project > user」
- **公式記述**: *"優先順位（高→低）：Managed > Command line > Local > Project > User"*
- **修正案**: 「優先順位（高→低）: managed > Command line（CLI 引数） > local > project > user」

## SUGGESTION（改善余地）

### S-01: 「配布インフラ」が本文で未説明

- **箇所**: 概要（L16）
- **記載内容**: 「実体は **Plugin 単体 + Marketplace カタログ + 配布インフラ** の三層構造」
- **問題点**: 「配布インフラ」が本文中で具体的に説明されていない（git ホスティング・自動更新機構等を指すと推測される）
- **修正案**: 「配布インフラ（git ホスティング・自動更新・キャッシュ管理）」のように三層構造の定義を補足

### S-02: `commands/` を「旧スキル」と呼ぶのはニュアンスが強い

- **箇所**: 1-2 Plugin の構造図（L45）
- **記載内容**: `├── commands/                # フラット形式の旧スキル（互換用）`
- **問題点**: 公式記述は *"flat Markdown files. Use `skills/` for new plugins"* であり、「旧」というニュアンスは含まれない
- **修正案**: `# フラット形式のスキル（新規は skills/ 推奨）`

### S-03: 「5名以下」は公式根拠なし

- **箇所**: 4-2「利用規模別の推奨アプローチ」表（L490）
- **記載内容**: 「個人または小規模チーム（**5名以下**）」
- **問題点**: 「5名」という具体数字に公式根拠なし。レビュアーの主観
- **修正案**: 「個人または小規模チーム（数名規模を目安）」など曖昧表現に変更

### S-04: JSON コード内コメントは厳密には構文エラー

- **箇所**: 3-2 (1) Managed Settings の JSON 例（L312-314）
- **記載内容**:
  ```json
  // macOS: /Library/Application Support/ClaudeCode/managed-settings.json
  {
    ...
  }
  ```
- **問題点**: 標準 JSON ではコメントを含められない
- **修正案**: コードフェンスを ` ```jsonc ` に変更するか、コメントをコードブロック外の本文に移動

### S-05: 「インストール済みバージョンの一覧表示 ❌ なし」は厳しすぎ

- **箇所**: 3-2 (9) 自動更新・バージョン追跡 表（L442）
- **記載内容**: 「インストール済みバージョンの一覧表示 | ✅ `/plugin` UI | ❌ なし」
- **問題点**: clone 方式でも `git log` で履歴閲覧は可能。「UI として一覧化されない」点を考慮すると △ が公平
- **修正案**: 「△ `git log` で確認可能（UI なし）」

### S-06: 「フックの配布 ✅ settings.json コピーで可能」は単純化しすぎ

- **箇所**: 3-3「代替手段でも同等に実現できること」表（L453）
- **記載内容**: 「フックの配布 | ✅ | ✅ `settings.json` コピーで可能 |」
- **問題点**: Plugin の hooks は `${CLAUDE_PLUGIN_ROOT}` 等の変数置換やプラグイン bin/ への参照が使えるため、単純なコピーでは完全に同等にならない
- **修正案**: 「△ `settings.json` コピーで可能（変数置換・bin/ 参照は不可）」

## POSITIVE（良い点）

- **P-01**: 4 Part 構造（理解 → 実装 → 比較 → 適用）が論理的で、読み手が知りたい順番に並んでいる
- **P-02**: 各機能比較が具体的なコード例付きで、実装イメージが湧きやすい
- **P-03**: 結論部の優先度マーキング（★★★/★★/★）が意思決定の判断材料として機能している
