## 概要

## 事項(1): プロンプト以外のインタラクション手段

**問題**: CLI プロンプトの欠点
- 複数行テキスト（ソースコード・ログ等）のコピー&ペーストができない（Enter で送信）
- 長文メッセージの推敲自由度が低い
- 入力・改行時の誤操作で中途半端な状態のメッセージを送信しやすい

### B. CLI 内での複数行入力改善

設定なし〜最小設定で即時対応できる手段。  
**出典**: https://code.claude.com/docs/en/terminal-config, https://code.claude.com/docs/en/interactive-mode

| 問題 | 解決手段 | 操作 | 出典 |
|---|---|---|---|
| Enter で誤送信 | **Ctrl+J** で改行のみ挿入（設定不要・全ターミナル） | `Ctrl+J` | [terminal-config.md: "Enter multiline prompts"](https://code.claude.com/docs/en/terminal-config#enter-multiline-prompts) |
| 複数行テキストのペースト | **`\` + Enter** で行継続（設定不要・全ターミナル） | `\` → `Enter` | [terminal-config.md: "Enter multiline prompts"](https://code.claude.com/docs/en/terminal-config#enter-multiline-prompts) |
| 長文の推敲 | **外部エディタで編集**してから送信 | `Ctrl+G`（`$EDITOR` で開く） | [interactive-mode.md: "General controls"](https://code.claude.com/docs/en/interactive-mode#general-controls) |
| 送信途中のやり直し | **Ctrl+S** でプロンプトを stash 保存（後で復元） | `Ctrl+S` | [keybindings.md: "Chat actions"](https://code.claude.com/docs/en/keybindings#chat-actions) |
| コード・ログの渡し方 | **`@filename` 参照構文**でファイル内容を渡す | `@path/to/file` | [common-workflows.md: "Reference files and directories"](https://code.claude.com/docs/en/common-workflows#reference-files-and-directories) |
| 大量テキストのペースト | 10,000 字超は自動圧縮して安全に送信 | そのままペースト | [terminal-config.md: "Paste large content"](https://code.claude.com/docs/en/terminal-config#paste-large-content) |
| 画像・スクリーンショットを渡す | クリップボードにコピー後 **chat:imagePaste** アクションでペースト。Claude Code が画像を認識して UI バグ等を分析 | macOS: `Cmd+Shift+Ctrl+4` → `Ctrl+V` / Windows: `Win+Shift+S` → **`Alt+V`** | [keybindings.md: "Chat actions"](https://code.claude.com/docs/en/keybindings#chat-actions) |

**Shift+Enter 改行対応状況**:

| 区分 | 対象端末 |
|---|---|
| 設定不要（ネイティブ対応） | Ghostty, Kitty, iTerm2, WezTerm, Warp, Apple Terminal |
| `/terminal-setup` 実行後に対応 | VS Code, Cursor, Windsurf, Alacritty, Zed |
| 非対応 | Windows Terminal, gnome-terminal, JetBrains 系（PyCharm/Android Studio 等） |

出典: [terminal-config.md: "Enter multiline prompts"](https://code.claude.com/docs/en/terminal-config#enter-multiline-prompts)

**`Ctrl+G`（外部エディタ統合）が最実用的**: プロンプト入力中に `Ctrl+G` → `$EDITOR`（VSCode, Vim 等）が開く → 自由に推敲後に保存 → 入力欄に反映。長文・コードを含む指示の推敲に最も有効。  
出典: [interactive-mode.md: "General controls"](https://code.claude.com/docs/en/interactive-mode#general-controls)

**Vim キーバインド**: `settings.json` に `{ "editorMode": "vim" }` を設定すると複数行編集が効率化する。  
出典: [terminal-config.md: "Edit prompts with Vim keybindings"](https://code.claude.com/docs/en/terminal-config#edit-prompts-with-vim-keybindings)

**キーバインドのカスタマイズ**: `~/.claude/keybindings.json` で `chat:newline`, `chat:submit` 等を任意のキーに割り当て可能。  
出典: [keybindings.md](https://code.claude.com/docs/en/keybindings)

セッション間の文脈ロスを減らし、別セッション・別端末からスムーズに作業を再開するための手段。

**セッション命名と再開**（公式機能）

| 操作 | コマンド / フラグ | 出典 |
|---|---|---|
| セッション開始時に名前を付ける | `claude -n "auth-refactor: plan-done"` | [cli-reference.md: --name/-n](https://code.claude.com/docs/en/cli-reference#cli-flags) |

命名パターン例: `[auth-refactor: plan-approved]` / `[bug-123: investigating]` / `[sprint-12: day-3]`（参考: `cloudnative-co/claude-code-starter-kit/commands/handover.md` の推奨パターン）

**`/handover` コマンド**（参考: `cloudnative-co/claude-code-starter-kit`）

`HANDOVER.md` を生成して次のセッションへ引き継ぐ構造化ドキュメント。内容:
- 現在の git 状態（ブランチ・最終コミット・未コミット変更）
- 完了済み作業 / 残タスク（優先順チェックリスト）
- 重要な判断事項・既知の問題
- 次のセッションが最初に読むべきコンテキストファイル一覧
- 推奨次アクション（`claude -r [session-name]` でそのまま再開できる形式）

生成後: `/compact` または新セッション開始を推奨。`/rename` でセッションに名前を付けておくと `claude -r <name>` で確実に復帰できる。

**コンパクションのタイミング最適化**（参考: `cloudnative-co/claude-code-starter-kit`）

`/compact` を自動発火に任せると作業の途中で実行される。以下の境界タイミングで手動実行すると効果的:

| タイミング | 理由 |
|---|---|
| 探索フェーズ完了後・実装開始前 | 調査コンテキストを整理してから実装を開始 |
| マイルストーン完了後 | フェーズ間でコンテキストをリセット |
| 大きなコンテキスト切替前 | 無関係なコンテキストを持ち込まない |

出典: `cloudnative-co/claude-code-starter-kit/skills/strategic-compact/`

## 事項(2): 中間成果物を安全に保護する方法

**問題の本質**: 調査レポート・計画書等の中間成果物は、
- 最終的には本流ブランチ（main）の Git 管理対象にしたくない
- しかし作業完了まで `git clean` やブランチ切り替えで失われないよう保護したい

### 前提: 現状の .gitignore アプローチは想定より安全

`.gitignore` 除外フォルダは以下から**保護されています**:
- `git checkout <branch>` → 無視ファイルは消えない ✅（[git-checkout](https://git-scm.com/docs/git-checkout) – untracked ファイルを保護）
- `git clean -fd` → 無視ファイルは消えない ✅
- `git stash` → 無視ファイルは対象外 ✅（[git-stash](https://git-scm.com/docs/git-stash) – untracked/ignored ファイルは対象外）

**危険なのは `git clean -fdx`（`-x` フラグ付き）のみ**。通常操作では失われない。  
出典: [git-clean](https://git-scm.com/docs/git-clean)（`-x` オプション: "Don't use the standard ignore rules"）

### 代替アプローチ比較

| # | アプローチ | 保護強度 | 手間 | 向いているケース |
|---|---|---|---|---|
| **現状** | `.gitignore` 除外フォルダ | 中（`-fdx` に弱い）| 最小 | ローカル完結・シンプル重視 |
| **①** | 孤立ブランチ (orphan) を remote に push | 高（remote backup） | 小 | 同一リポジトリでバックアップしたい・端末間共有したい |
| **②** | `.claude/reports/` を独立 git リポジトリ化 | 最高（親リポジトリと完全独立） | 中 | 独立性最優先・長期保存 |
| **③** | git worktree で別ディレクトリに隔離 | 高（物理的に分離） | 中 | 複数ブランチを同時に参照したい |
| **④** | クラウド同期（OneDrive/Dropbox）と併用 | 高（クラウドバックアップ） | 最小 | すでにクラウド同期フォルダ内にある場合 |
| **⑤** | `PreCompact` フック + 自動コミット | 中（フック発火タイミング依存） | 最小（設定一度のみ） | セッション内で `/compact` 前に成果物を自動コミットしたい（①〜④と**併用**） |

### 判断フロー

```
C:\workspace\ がすでにクラウド同期フォルダ内にある?
├─ Yes → 現状維持（.gitignore 除外）+ クラウドが自動バックアップ ← ④
└─ No  → remote backup / 端末間共有が必要?
         ├─ Yes → ① orphan branch push（最もシンプル）
         └─ No  → git clean -fdx を誤実行するリスクが高い?
                  ├─ High → ② 独立リポジトリ（親リポジトリ操作と完全隔離）
                  └─ Low  → 現状維持（.gitignore 除外）で十分
```

### 各アプローチ 詳細

#### ① 孤立ブランチ（orphan）を remote に push【推奨】

```bash
# 作業開始時: orphan ブランチを作成して成果物を commit・push
git checkout --orphan work-notes/feature-name
git rm -rf --cached .          # index から既存ファイルを除外
git add .claude/reports/
git commit -m "wip: intermediate artifacts for feature-name"
git push origin work-notes/feature-name

# 作業完了後: main に戻る（orphan ブランチはそのまま or 削除）
git checkout main
git branch -D work-notes/feature-name       # ローカル削除（remote は残る）
git push origin --delete work-notes/feature-name  # remote も削除する場合
```

- 同一リポジトリの remote に push → **バックアップ確保 + 端末間共有可能**
- main ブランチ履歴とは**完全に切り離された孤立ブランチ**（`git log` に混入しない）
- 作業完了後にブランチ削除で整理可能  
出典: [git-checkout --orphan](https://git-scm.com/docs/git-checkout#Documentation/git-checkout.txt---orphanltnew-branchgt)

#### ② `.claude/reports/` を独立 git リポジトリ化

```bash
git init .claude/reports/
cd .claude/reports/
git remote add origin <private-remote-url>
git add .
git commit -m "initial: intermediate artifacts"
git push
```

- `.claude/reports/` が独自 git 管理 → **親リポジトリの操作に一切影響されない**
- 子ディレクトリに `.git` があると親リポジトリは中身を追跡しないが、`git add` 時に "adding embedded git repository" 警告が出る。`.gitignore` 併用が無難 `[未確認]`: 「自動無視」を直接裏付ける公式仕様  
出典: [gitignore](https://git-scm.com/docs/gitignore)

#### ③ git worktree で別ディレクトリに隔離

```bash
# work-notes ブランチを ../project-docs ディレクトリとして checkout
git worktree add ../project-docs work-notes
# ../project-docs/ 内で成果物を管理（main ディレクトリの git 操作に影響されない）
```

- 同一リポジトリの別ブランチを別ディレクトリで操作
- main のディレクトリと物理的に分離  
出典: [git-worktree](https://git-scm.com/docs/git-worktree)

#### ④ クラウド同期と .gitignore の組み合わせ（最小手間）

`C:\workspace\` が OneDrive や Dropbox の同期対象フォルダ内にある場合、`.gitignore` 除外でもクラウドが自動バックアップする。追加設定不要。

#### ⑤ PreCompact フック + 自動コミット（セッション内の自動保護・他の手段と併用）

```json
// .claude/settings.json または ~/.claude/settings.json の hooks 設定
{
  "hooks": {
    "PreCompact": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "cd \"$CLAUDE_PROJECT_DIR\" && git add -A && git diff --cached --quiet || git commit -m 'checkpoint: pre-compact auto-commit'"
          }
        ]
      }
    ]
  }
}
```

- `/compact`（自動・手動ともに）の直前に `PreCompact` フックが発火 → 変更を自動コミット
- 長いセッションで `/compact` が走っても、その時点の成果物がコミット済みになる
- **Git 管理下のファイルのみ対象**。`.gitignore` 除外ファイルの保護には ①〜④ を別途組み合わせること  
フック仕様: [hooks.md: PreCompact](https://code.claude.com/docs/en/hooks#precompact)（公式確認済み）  
参考実装: `cloudnative-co/claude-code-starter-kit/features/pre-compact-commit/hooks.json`

### 補足: 名前付きチェックポイントと Safety Net

#### `/checkpoint` コマンド（参考実装）

参考実装: `cloudnative-co/claude-code-starter-kit/commands/checkpoint.md`

```bash
/checkpoint create "feature-start"   # 名前付きで git stash/commit + ログ記録
/checkpoint verify "feature-start"   # ベースラインとの差分確認（変更ファイル数・テスト通過率）
/checkpoint list                      # 全チェックポイントを日時・SHA 付きで一覧
```

`.claude/checkpoints.log` に `日時 | チェックポイント名 | git SHA` を記録。`verify` で変更ファイル数・テスト通過率の比較が可能。  
**注意**: これは cloudnative-co のスラッシュコマンド実装。Claude Code 公式の `/checkpoint` コマンドではない。

#### Safety Net フック（破壊的操作のブロック）

参考実装: `cloudnative-co/claude-code-starter-kit/features/safety-net/`

```json
{
  "env": { "SAFETY_NET_STRICT": "1" },
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{"type": "command", "command": "cc-safety-net --claude-code"}]
    }]
  }
}
```

`PreToolUse` フックで以下の破壊的コマンドを実行前にブロック: `git reset --hard` / `git checkout -- <file>` / `git push --force` / `rm -rf`  
`SAFETY_NET_STRICT=1`（デフォルト）: パース不能なコマンドも fail-closed でブロック。  
要件: `npm install -g cc-safety-net`  
フック仕様: [hooks.md: PreToolUse](https://code.claude.com/docs/en/hooks)（公式確認済み）

## 参考: 成果物ライフサイクル管理

### Git 管理 vs .gitignore 除外の判断基準

**大原則**: 「チーム共有が必須・再利用可能なもの」→ Git 管理。「セッション固有・個人環境差異があるもの」→ `.gitignore` 除外。  
出典: 調査報告書 `.claude/reports/design-base-kit/claude-code-research/2026-04-24_調査B_サブフォルダ構造ベストプラクティス.md`, `cloudnative-co/claude-code-starter-kit/.gitignore`

| ファイル・フォルダ | 管理方針 | 理由 |
|---|---|---|
| `.claude/settings.json` | ✅ Git 管理 | チーム共有が必須（パーミッション・hooks） |
| `.claude/rules/` | ✅ Git 管理 | チーム共有の規約 |
| `.claude/skills/` | ✅ Git 管理 | 再利用可能なコマンド |
| `.claude/agents/` | ✅ Git 管理 | 共有エージェント定義 |
| `CLAUDE.md` | ✅ Git 管理 | プロジェクト共通指示 |
| `.env.example` | ✅ Git 管理 | 秘匿値なしのテンプレート |
| `.claude/settings.local.json` | ❌ .gitignore 除外 | OS 固有・個人用設定 |
| `CLAUDE.local.md` | ❌ .gitignore 除外 | 個人用補足指示 |
| `.claude/reports/` | ❌ .gitignore 除外（原則） | セッション成果物・環境差異あり |
| `.claude/work/` | ❌ .gitignore 除外 | セッション一時ファイル |
| `.env` | ❌ .gitignore 除外 | 秘匿情報（API キー等） |

### 成果物の「昇格」パターン

セッション成果物（`.claude/reports/` 配下）から得た知見は、再利用可能な形に変換して Git 管理対象に昇格させる:

```
.claude/reports/<activity>/           ← セッション中の調査・レビュー記録（.gitignore 除外）
    ↓ 知見を抽出・一般化
CLAUDE.md                             ← プロジェクト全体指示に反映（Git 管理）
.claude/rules/<name>.md              ← 再利用規約として定義（Git 管理）
.claude/skills/<name>/SKILL.md       ← 再利用コマンドとして実装（Git 管理）
```
