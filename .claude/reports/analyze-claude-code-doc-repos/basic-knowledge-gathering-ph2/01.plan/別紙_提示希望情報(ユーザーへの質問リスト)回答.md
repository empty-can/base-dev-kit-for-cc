### A. スコープに関する基本方針

- **Q1**: 既clone済みリポジトリ（claude-plugins-official / claude-ai-mcp / servers / cloudnative-co/claude-code-starter-kit）は今回の選定で「重複扱い（再評価不要）」と「再評価対象」のどちらか？

- **Q3**: 3団体以外で、リポジトリの README から発見した別組織のリポジトリを「関連候補」として推薦してよいか？

      ただし、公式団体のものではないという点から内容の信頼性や正確性、時によってはセキュリティ面で、公式のそれに比べて劣るリスクがあることも留意しておく必要はあります。具体的には、そのような団体のGitHubのHomeページには、AnthropicやModel Context Protocolのページには表示されている「verified」マークが表示されていません。  
      もっとも、これは Cloud Native inc. の`cloudnative-co/claude-code-starter-kit`からナレッジを収集する際に既に留意済みの観点でもあるため、今回もそれに準じるという形です。

#### G1. 「小規模なローカルRAG」の判定基準（Q4 例外条項の運用化）

最高優先扱いするには、選定フェーズで判定可能な定義が必要です。以下の観点でご回答ください。

  - (d) すべて
- **回答** 
  → (c)  
    - (a)ではあらかじめデータ構造を十分に設計しないと検索自由度が上がらず、またそもそも仕組みに由来する構造的限界があると、この仕組みを構築した際にClaudeからコメントをもらっています。

  - (b) トークン効率（検索結果の精度・コンパクトさ）
  - (c) Claude Code との統合容易さ（hook / skill / mcp 経由で組める）
  - (d) すべて重視
- **回答**  
  → (b) >>> (c)

  - (c) README 全文 + CLAUDE.md（あれば）+ トップディレクトリ構造、それ以上は深追いしない
  - (d) その他
- **回答**  
  → (a)、(a)で推薦するか否かを判定できなかった場合は(b)まで。  
    利用者向けに書かれた`README.md`の内容で推薦するか否か判断できなかったリポジトリの`CLAUDE.md`に、判断に足る情報が書かれている可能性は低いように思います。（`CLAUDE.md`はClaude向けに書かれたドキュメントである認識なので）

- **G5-b**: 「打ち切り後にもう少し情報が欲しい」と感じた候補の扱い
  - (a) 一旦保留扱いで `confidence: low` にし、ユーザー側で個別深掘り依頼が来たら追加調査
  - (b) その場で深掘りする
- **回答**  
  → (a)

#### G6. 採用しなかった候補の記録粒度

- **G6-a**: 却下リストの記録レベル
  - (a) URL ＋ 一行却下理由のみ
  - (b) URL ＋ 却下理由 ＋ 取得済みメタ情報（star/fork/最終更新等）
  - (c) URL ＋ 却下理由 ＋ メタ情報 ＋ 簡易的な内容サマリ
  - → 推奨は (b)。後日「なぜ採用しなかったか」を再評価できる最低限の情報
- **回答**  
  → (b)  
    - 簡易的な内容サマリを書きたくなったとしたら、それは恐らく却下の根拠としたメタ情報が実は却下の根拠として弱く適切でない可能性があり、判断そのものを見直すべきと思われます。

- **G6-b**: 「ボーダーライン候補」（採用すべきか即断できなかったもの）の扱い
  - (a) 推奨側に含めて `confidence: low` のフラグ
  - (b) 却下側に含めて「再考可能」コメント
  - (c) 専用の「保留候補」リストを別ファイルで作成
- **回答**  
  → (c)  
    - 活動を進めていく中で、いつ保留から推奨へ昇格/却下へ降格の判断にできる情報が得られるか分からないため完全に独立したリストファイルとして管理しておきたいです。

- **回答**

- **G7-b**: 非 verified 組織のリポジトリを推薦する際の追加注記

  - (b) 個別判断（特筆すべき点があれば記載）
  - (c) 不要
- **回答**

    - (a) は、固定文言を機械的に記載しても、verifiedのフラグ列以上の情報量が無いので無駄と考えます。
